# Plan: Render GVC Sandbox Homepage in React

## Background

The GVC sandbox lives at `client/src/lib/sandbox/index.html`. It is a standalone HTML page that:
- Loads jQuery from `../lowcode/glitterBundle/jquery.js` (the bundle does not yet exist in this repo)
- Loads `GlitterInitial.js` (ES module) which bootstraps the GVC runtime
- Requires two DOM nodes: `#glitterPage` and `#Navigation`
- Calls `glitter.setHome('./pages/home.js', 'home', {})` to load the home page module

The goal is to have the React app (`App.tsx`) render this GVC homepage inline, without an iframe.

---

## Key Technical Insight: Path Resolution

`PageManager.js` computes page URLs at runtime via:
```js
static getRelativeUrl(url) {
  return new URL(url, new URL('../../', import.meta.url).href).href;
}
```

If `PageManager.js` is served from `/lowcode/glitterBundle/module/PageManager.js`, then:
- base = `new URL('../../', ...)` = `/lowcode/`
- `./pages/home.js` → `/lowcode/pages/home.js`

This means **all GVC runtime files must be served as static assets** (from `public/`) at predictable URLs that Vite does not transform. This avoids breaking `import.meta.url` calculations and dynamic script injection inside the GVC runtime.


### Step 3 — Create `GvcPage.tsx`

Create `client/src/gvc-pages/GvcPage.tsx`:

```tsx
import { useEffect } from 'react';

export function GvcPage() {
  useEffect(() => {
    // Prevent double-initialization (React StrictMode runs effects twice in dev)
    if ((window as any).__gvcInitialized) return;
    (window as any).__gvcInitialized = true;

    // Step 1: load jQuery globally, then bootstrap GVC
    const jqueryScript = document.createElement('script');
    jqueryScript.src = '/lowcode/glitterBundle/jquery.js';
    jqueryScript.onload = () => {
      // Step 2: load GlitterInitial as an ES module
      const initScript = document.createElement('script');
      initScript.type = 'module';
      initScript.src = '/sandbox/GlitterInitial.js';
      document.head.appendChild(initScript);
    };
    document.head.appendChild(jqueryScript);
  }, []);

  return (
    <>
      <div id="glitterPage" />
      <div id="Navigation" />
    </>
  );
}
```

**Why script injection instead of `import()`?**  
The GVC runtime uses `import.meta.url` internally to compute asset URLs. If Vite transforms these modules during the build, the URLs change and `getRelativeUrl` breaks. Injecting `<script>` tags bypasses Vite's module graph, serving files directly from `public/` with stable URLs.

### Step 4 — Update `App.tsx`

```tsx
import { GvcPage } from './gvc-pages/GvcPage';

function App() {
  return <GvcPage />;
}

export default App;
```

## Caveats / Considerations

1. **StrictMode double-run**: The `__gvcInitialized` guard prevents double-initialization when React 18 StrictMode calls `useEffect` twice in development.

2. **CSS**: The sandbox `index.html` referenced no extra stylesheets, but `glitterBundle` includes `Glitter.css` and `bootstrap.css`. If the GVC UI looks unstyled, those can be loaded by adding `<link>` tags in `useEffect` or in `index.html`.

3. **Page-level CSS (`html` scroll)**: `GlitterInitial.js` manipulates `document.querySelector('html').scrollTop`. Make sure the React app's root container (`#root`) does not have `overflow: hidden` that blocks this.

4. **Routing coexistence**: The `react-pages/` and `gvc-pages/` directories suggest a future where React Router handles navigation between React pages and GVC pages. This plan only wires up the home page; routing can be added later.

5. **Production build**: For `vite build`, the `public/` assets are copied as-is, so the paths remain correct. No additional Vite config changes are needed.
